# TL;DR

Azure AI Speech Fast Transcription, evaluated on 5 YouTube interviews (~9 hours of audio total), each tested in two audio formats (16 kHz WAV PCM and 32 kbps Opus), each run both clean and with office babble overlaid at 15 dB SNR. That's 20 cloud calls, ~$5 on Azure's Standard tier.

Condition Mean WER 95% CI Target Pass?
Clean audio (WAV + Opus combined) 7.23% 6.05% – 8.47% 20% PASS
+ 15 dB office babble 8.54% 7.03% – 10.09% 25% PASS

Three things showed up that a naïve benchmark wouldn't catch:

  1. Opus (32 kbps, voice-tuned) is statistically tied with WAV on clean audio — format compression buys the speed wins without paying WER cost.
  2. Opus actually performs better than WAV under office noise (−0.14pp mean, 4 of 5 videos). The codec's perceptual model quietly de-emphasizes non-speech spectral content, acting as a mild noise filter before the cloud STT ever sees the audio.
  3. The noise penalty isn't uniform — videos with lower-energy speech (whispered asides, soft-spoken guests) take 2–3× the WER hit of louder, studio-mic content.

# Why this exists

Most WER benchmarks are run on clean, read-aloud audiobook text — LibriSpeech, mostly. Real speech-to-text users aren't reading audiobooks. They're dictating voice memos on phones, interviewing people, recording meetings in rooms with HVAC noise.

When I needed to answer format-comparison and noise-robustness questions for a voice-capture feature, I couldn't find a drop-in harness that handled all of: real ground-truth sourcing, WAV-vs-Opus format isolation, SNR-calibrated noise injection, and honest statistical gating. So I built one.

This repo is the thing I wish had existed. It's small (~1000 LoC of pipeline), swappable at the backend (replace transcribe.py for any STT provider), and opinionated about the parts that matter: honest normalization, CI-upper-bound pass/fail, and atomic idempotent caching so you can re-run without duplicate cloud bills.

# Methodology

What a WER pipeline actually needs to get right, and how this one handles each piece.

# Ground truth from YouTube creator captions

YouTube surfaces two caption tracks per video:

  • Available subtitles — uploaded by the creator, usually time-aligned to the audio and near-verbatim. This is real ground truth.
  • Available automatic captions — generated by Google's own ASR. Not ground truth, but a useful pseudo-reference for directional signal.

Creators with accessibility-conscious audiences (MKBHD, Vox, TED, most educational channels) upload their own captions on every video. yt-dlp --list-subs distinguishes the two. The pipeline auto-detects which track it pulled and labels every clip in the final report with ref_source: creator or ref_source: auto.

Why not use LibriSpeech?

LibriSpeech is 1000 hours of audiobook narrations read aloud by single speakers into studio microphones. It's a benchmark dataset, not a usage dataset. Modern STT models score 2–3% WER on LibriSpeech test-clean. Those same models can score 15–30% WER on a meeting recording. The measurements aren't comparable.

YouTube interviews give you multi-speaker conversational English with real acoustic conditions (some studio, some remote, some background music) and verbatim creator captions — a far better proxy for how voice-capture features actually get used.

# WAV vs Opus — apples-to-apples

The format comparison test is simple in concept but easy to get wrong in practice. For each video, the pipeline:

  1. Downloads the source M4A audio from YouTube (no re-encoding needed — YouTube already serves AAC).
  2. Transcodes to 16 kHz mono PCM_16 WAV via bundled ffmpeg (lossless decompression of the AAC).
  3. Transcodes the same source to Opus @ 32 kbps in OGG container, using libopus with application=voip (voice-tuned encoding at narrow-band).
  4. Ships both files to Azure Fast Transcription as separate requests.
  5. Scores each hypothesis against the same creator-caption reference.

Both files are derived from the same source, so any WER delta is attributable to the codec, not to input differences. No double-encoding: we never take a compressed file and re-encode it to another compressed format.

WAV (16 kHz PCM) Opus (32 kbps, voice-tuned)

Finding: Opus is not a WER liability

Mean delta across 5 videos: +0.09 pp (Opus vs WAV). Confidence intervals overlap almost completely (WAV CI 5.61–8.75, Opus CI 5.52–9.11). For STT evaluation purposes, 32 kbps Opus and 16 kHz WAV are interchangeable on clean audio.

# Noise injection — real SNR math

The pipeline ships a noise injector (noise.py) that does one thing correctly: mix a noise signal into a speech clip at a target signal-to-noise ratio, then verify the achieved SNR on the mixed output.

The math

For a target SNR in dB:

RMS(noise_scaled) = RMS(speech) / 10^(SNR_dB / 20)
mixed             = speech + noise_scaled
# If peak > 0.99, scale BOTH components by same factor (preserves SNR ratio)

Two common mistakes the pipeline avoids:

  • Peak-normalizing the mix after injection. Tempting, but it scales speech and noise by the same factor, and breaks the "achieved SNR" measurement because the post-mix signal no longer decomposes cleanly into original-speech + noise-component. Fix: scale both components pre-save by a shared factor when peak exceeds 0.99, and measure SNR from the scaled components directly.
  • Pre-scaling speech to an unrealistic RMS level. An earlier iteration of this code pre-scaled speech to −3 dBFS RMS (≈0.707) on the theory that it would normalize levels across clips. Real speech has a 10–20 dB crest factor, so peaks immediately hit ±2.0–3.0 and the clipping guard kicked in hard, dropping the effective SNR to ≈1 dB instead of the targeted 15. Fix: no pre-scaling. Use the source's natural level.

Office babble source

The default noise file is a synthetic "office ambient" mix: three overlapping podcast-speaker WAVs at low volume (speech-shaped babble, non-overlapping speaker IDs so Azure can't coherently transcribe the background) plus low-level pink noise at roughly −18 dB for HVAC-like ambient. If you drop your own recordings into data/noise/*.wav, the pipeline uses those instead. If nothing is present, it falls back to 1/f pink noise — better than white Gaussian but still labeled "synthetic" in every report that uses it.

Finding: Opus is slightly MORE robust to noise than WAV

Under 15 dB office babble, WAV degrades by +1.42 pp on average (7.19% → 8.61%), but Opus degrades by only +1.20 pp (7.27% → 8.47%). In 4 of 5 videos, Opus was more robust. The 32 kbps voice-tuned encoder's perceptual model attenuates non-speech spectral content before the cloud STT decodes it — effectively a free, lightweight noise pre-filter.

Finding: Noise impact scales with speech energy, not just SNR

The largest WER hit came from an interview with a soft-spoken guest using lots of whispered asides — WAV +3.06 pp, Opus +1.93 pp. Louder, studio-mic interviews took only 0.7–1.3 pp. Takeaway: "15 dB SNR" is not a sufficient description of acoustic conditions. The metric that matters is the per-segment energy of the target speaker relative to the babble, which varies within a single clip.

# Text normalization — symmetric or bust

jiwer's default tokenization counts "I'm" vs "I am" as a substitution-plus-insertion. It counts "1" vs "one" similarly. It penalizes the STT for "offsite" when the creator caption wrote "off site". None of those are real transcription errors — they're normalization artifacts.

The pipeline applies a symmetric normalization pass to both reference and hypothesis before scoring:

  1. Strip speaker labels of the form Speaker Name (HH:MM:SS):
  2. Strip bracketed annotations like [inaudible 00:00:42]
  3. Lowercase
  4. Expand common contractions (a small curated map, 40+ entries)
  5. Strip punctuation except apostrophes (those get stripped next)
  6. Strip apostrophes
  7. Convert standalone digits to words (0–19, 20/30/…/90, 100, 1000, 1000000)
  8. Remove a conservative set of disfluencies (um uh uhm hmm mhm ah er eh). Notably we don't strip "like" / "you know" / "I mean" since those are often real content.
  9. Collapse whitespace

Applied to both sides. This is not the same as Whisper's full English normalizer (which does inverse text normalization for currencies, ordinals, etc.) — but it's good enough for conversational speech and entirely self-contained (no external language model).

# Statistical gating — CI upper bound, not point estimate

At N=5 videos, a point-estimate mean of 4.9% vs a 5% target means essentially nothing. The true mean might be 3%, might be 7% — the data is too sparse to tell.

So the pipeline reports bootstrap 95% confidence intervals on the mean (1000 resamples) and gates pass/fail on the upper bound of the CI. 

pass_fail = "PASS" if ci_upper <= target else "FAIL"

This is the conservative call. At small N, a clean point-estimate win can still FAIL if the CI is wide. As N grows, CIs tighten and the rule becomes less conservative naturally. No need to hand-tune a significance threshold.

Every per-scenario report also includes median + IQR, per-clip WER, and the top-3 worst-performing clips with their normalized reference/hypothesis pairs — the fastest way to check whether a regression is a systemic quality issue or a single pathological clip.

# Quickstart

The pipeline runs on Python 3.13 or 3.14, Windows/macOS/Linux. Bundled ffmpeg binary via imageio-ffmpeg means no admin install. 

git clone https://github.com/shivachintaluru-prog/stt-wer-eval.git
cd stt-wer-eval
pip install -r requirements.txt

# Provision an Azure AI Speech resource — F0 free tier gives 5 audio-hrs/month.
# https://ai.azure.com/  or  portal.azure.com → Create → Speech
export AZURE_SPEECH_KEY='your-32-char-key'
export AZURE_SPEECH_REGION='eastus'

# Dry-run — preview clip enumeration + estimated audio minutes, zero API calls.
python run.py --tier multi_speaker --max-clips 2 --dry-run

# Real run — WAV + Opus variants per video (2 Azure calls per video).
python run.py --tier multi_speaker --max-clips 2

# With noise — office babble injected at 15 dB SNR on top of the clean variants.
python run.py --tier multi_speaker --max-clips 5 --noise-snr 15

Reports land in data/<scenario>/reports/report_<timestamp>.md with headline, per-clip breakdown, format comparison tables, and clean-vs-noisy deltas. Raw Azure response JSONs go in data/<scenario>/hypotheses/ — these are the durable artifact. Re-running is idempotent (skips any clip with a valid cached hypothesis).

# Architecture

run.py                  Orchestrator + CLI
├── config.py           Scenarios, targets, YouTube list, endpoints
├── io_utils.py         Atomic writes, secret redaction, disk guard, logging
├── dataset.py          Loaders: user_clips, YouTube captions, Azure TTS
├── transcode.py        ffmpeg wrapper (WAV + Opus, bundled binary)
├── noise.py            SNR-calibrated noise mixer + pink-noise fallback
├── transcribe.py       Azure Fast Transcription REST client  ← swap me
├── wer.py              Symmetric normalization + jiwer + bootstrap CI
└── report.py           Per-scenario + combined Markdown reports

Each stage reads/writes to disk and is idempotent. Ctrl-C is safe. Re-running only touches Azure for clips that don't already have a validated hypothesis JSON. Swap transcribe.py for any other STT provider and the rest works unchanged.

# Cost & safety defaults

  • Hard cap on cloud calls per run (MAX_TRANSCRIPTIONS_PER_RUN = 50) — aborts loudly rather than silently billing thousands.
  • --dry-run enumerates what would be sent (clip count, estimated audio minutes) without touching the API. Always run this first.
  • Retry with Retry-After honored on 429s. Retry on 422s too (Azure occasionally returns a transient "empty audio file" on multipart POST — goes away on retry).
  • Subscription keys redacted in every logging path. io_utils.redact_headers() is called on both request and response headers before anything hits stdout or the log file. No API key ever lands on disk.
  • Atomic writes (tmp + fsync + rename) for all artifacts, with validation-on-skip. A half-written JSON from a killed process won't pass the validation check, so the next run re-does just that clip.
  • Per-request timeout 1800s — generous enough to cover uploading a 500 MB / 5-hour audio file plus Azure's server-side processing.

Representative cost: 20 Azure calls across ~9 hours of audio (five 15-60 min videos × WAV/Opus × clean/noisy) ran about $5 on Standard S0, with the first 5 hours covered by F0 free tier.

# Roadmap

  • More STT backends: Whisper (local + API), AWS Transcribe, Google STT, AssemblyAI, Deepgram.
  • Bitrate sweep within Opus (8 / 12 / 16 / 24 / 32 / 48 kbps) to map the quality-vs-size curve.
  • Re-enable HuggingFace research datasets (TED-LIUM, AMI, Common Voice) once the Python 3.14 / datasets / torchcodec compatibility shakes out.
  • MUSAN noise subset downloader (currently you BYO noise files or fall back to pink noise).
  • Paired bootstrap test for format deltas (same reference clip, WAV vs Opus → tighter CIs than independent bootstrap).
  • Diarization accuracy metrics (speaker identification F1) as a separate reported axis.
